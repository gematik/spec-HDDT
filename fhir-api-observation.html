Endpoint Observation

Seiteninhalt:

  • Endpoints
  • Examples

DiGAs query the resource server of a Device Data Recorder via standard RESTful APIs. All device data measured by a Personal Health Device is represented as FHIR Observation resources at the resource server.

HDDT defines specific profiles for Observation resources for each Mandatory Interoperable Value (MIV). This allows for considering specific requirements of typical DiGA use cases and fosters the adoption of already existing profiles. In addition, aggregated values for a specific MIV can be made available as condensed reports via HDDT-defined FHIR Operations. See:

  • MIV-specific APIs for a list of all MIV-specific Observation profiles and operations,
  • Information Model for a detailed description of elements and codings which are common to all MIV-specific Observation profiles.
  • Retrieving Data for a detailed description of the overall data flow and how relationships between Observation, DeviceMetric and Device resources are affected by the calibration of a sensor or the exchange of a sensor.

Nevertheless, all Observation profiles share common characteristics and the same basic API interactions. This chapter describes these generic interactions in detail. For a better understanding of the overall data flow and the difference between dedicated and continuous measurements, refer to chapter Retrieving Data.

To comply with § 374a SGB V, every Device Data Recorder MUST implement an Observation endpoint that allows for:

  • reading individual Observation resources.
  • searching for Observation resources associated with the current patient and an authorized MIV.

Endpoints

Observation - READ

Per HL7 FHIR, a resource is a definable, identifiable, addressable collection of data elements that represents a concept or entity in health care. By this definition, each Observation resource a Device Data Recorder transmits to a DiGA MUST be identifiable and addressable through a logical id. A Device Data Recorder MUST allow a DiGA to request a known Observation using a FHIR read interaction.

   
Endpoint /Observation/<id>
HTTP Method GET
Interaction READ
Description Retrieve a single Observation resource by its logical id. The returned Observation resource MUST conform to the MIV-specific Observation profile. All constraints and obligations of the standard FHIR read interaction apply.
Request Parameters id - logical ID of the Observation.
Authorization OAuth2 Bearer token REQUIRED. The resource server MUST only return an Observation resource that is linked to the patient who is associated with the token. Provided Device resources MUST match the requestor’s granted scopes (see Smart Scopes).
Returned Objects MIV-specific HDDT Observation resource.
Specifications The only way for a DiGA to obtain the logical id of an Observation resource is a ‘search’ interaction for device data of the given patient (see searching below).
A Device Data Recorder MAY limit access to historical data (e.g., only up to 30 days into the past). In case a DiGA requests such a historic resource by its id after the availability period ended, the Device Data Recorder MUST respond with a 404 Not Found error. A DiGA can discover the length of this period through the Historic-Data-Period property of the definition of the associated Device resource.
If a requesting DiGA can only process high-quality device data, it SHOULD assess the calibration status of the device, which can be obtained through the DeviceMetric resource accessible through the Observation.device reference. Important: If the Observation.device reference links to a Device resource instead of a DeviceMetric resource, the DiGA MUST consider the device to be calibrated without any need for re-calibration (see Calibration Logic for details).
Error codes 400 (Bad Request), 401 (Unauthorized), 404 (Not Found), 500 (Internal Server Error)

See Error Codes for complete error response formats, headers, and detailed scenarios.

Observation - SEARCH

Implicit Arguments

When a DiGA queries device data from a Device Data Recorder using the FHIR search interaction on the Observation resource type, the request header MUST contain a valid Access Token according to the HDDT OAuth2 profile.

The Device Data Recorder MUST extract two pieces of information from the Access Token:

  1. Patient Identifier: The Device Data Recorder MUST be able to discover the internal patient identifier from the access token. This identifier MUST be implicitly added as the subject parameter to every search query. If a DiGA explicitly provides a subject parameter, the Device Data Recorder MUST ignore it and SHOULD respond with a 400 Bad Request error.

  2. SMART Scope: The scope(s) accepted during pairing define which Mandatory Interoperable Values (MIVs) the DiGA may access. These scopes MUST be implicitly added as code parameters with OR-semantics. A DiGA MAY further narrow the search by providing additional code parameters.

For detailed guidance on exceptional cases, see Retrieving Data.

Generic HDDT Search Interaction
   
Endpoint /Observation
HTTP Method GET / POST
Interaction SEARCH
Description Search for Observation resources.
Authorization OAuth2 Bearer token REQUIRED. The Device Data Recorder acting as the resource server MUST only return Observation resources that are linked to the patient in the token and that are accessible by the requestor according to its scopes (see Smart Scopes for the compartment semantics that MUST be used for validating the authorization of the requestor).
Search Parameters A full list of the search parameters that this endpoint MUST support can be found on page FHIR Resource Server - Search Parameters.

• The Device Data Recorder MUST be able to discover the patient identifier from the Access Token. This identifier MUST implicitly be considered the subject parameter for every search request. If a DiGA explicitly provides a subject parameter with a query, the Device Data Recorder MUST ignore this parameter and SHOULD respond with a 400 Bad Request error.
• If a code argument is provided with the request, the search is narrowed to the provided codes. If one or many of the provided code values are not included in the MIV’s ValueSet, the Device Data Recorder MUST NOT return any matches for these code values. The Device Data Recorder MAY include an OperationOutcome in the search set Bundle to signal which requested code values were not found in the MIV’s ValueSet.
The Device Data Recorder MAY support additional standard FHIR Observation search parameters.
• A Device Data Recorder MAY limit access to historical data (e.g., only up to 30 days into the past). In case the date argument results in a search that only covers deleted historic data, the Device Data Recorder MUST return an empty Bundle and MAY additionally include an OperationOutcome to signal the DiGA that there may have been data which is no longer available.
Returned Objects Bundle containing Observation entries that match the query. Optionally, Device or DeviceMetric resources can be included in the Bundle if requested via an _include=Observation:device search parameter.
If no matching Observation resources are found, an empty Bundle MUST be returned.
Specifications The Device Data Recorder SHOULD support Paging.
The Device Data Recorder MUST consider the delay from real time of the specific sensor and Device Data Recorder. This device-specific value can be obtained as the Delay-from-real-Time property of the definition of the associated Device resource.
Data MAY be missing due to broken connectivity (e.g., the sensor is out of range of its mobile app). A DiGA can discover such situations by checking the status of the associated device.
Device Data Recorders MUST support _include=Observation:device and _include:iterate=DeviceMetric:source as query arguments.
For additional information on the constraints and obligations listed above, refer to chapter Retrieving Data.
Error codes 400 (Bad Request), 401 (Unauthorized), 500 (Internal Server Error)

See Error Codes for complete error response formats, headers, and detailed scenarios.

Calibration Logic

This specification distinguishes between two scenarios for device calibration:

  1. Devices requiring calibration: If a device requires periodic calibration (e.g., glucometers), the Observation.device element MUST reference a DeviceMetric resource. The DiGA MUST check the calibration.state in this resource to determine if measurements are valid.

  2. Factory-calibrated devices: If a device is factory-calibrated and requires no user calibration (e.g., some rtCGM systems), the Observation.device element MAY reference a Device resource directly. In this case, the DiGA MUST assume the device is calibrated.

Important: The decision to reference Device vs. DeviceMetric is made by the Device Data Recorder based on the device’s calibration requirements. DiGAs MUST handle both cases appropriately.

Searching for Dedicated Measurements

Dedicated measurements are scheduled according to a care plan or triggered by unscheduled events (e.g., when a patient feels unwell). A dedicated measurement occurs at a specific point in time and records single values for one or more data items (e.g., systolic and diastolic blood pressure). Typical examples of devices used ad hoc or based on a care plan are blood glucose meters, smart insulin pens, blood pressure cuffs, and peak flow meters.

For HDDT Observation Profiles that cover dedicated measurements, there are no deviations from the generic search interaction defined above.

Searching for Continuous Measurements

With continuous measurements, the patient is connected to a sensor (almost) all the time, and the sensor continuously measures and records data. Typical examples of devices that perform continuous measurements are rtCGM, insulin pumps, and pacemakers. In addition, there are devices and scenarios which perform continuous measurements over dedicated periods, e.g., a personal ECG recorder for recording short, single-channel ECGs or a pulse oximeter that monitors SpO₂ during sleep time. With respect to the flavor of the search interaction, such scenarios are usually considered as continuous measurements too, because this search flavor is much more efficient in transferring sampled data.

For HDDT Observation Profiles that cover continuous measurements, there are some constraints and obligations with respect to the generic search interaction defined above:

  • Values from continuous measurements MUST be provided as chunks of sampledData, where each chunk of data is represented as an Observation. Chunks are of fixed size with respect to the effectivePeriod. For example, a chunk with a fixed size of one hour can hold up to 60 values with a sampling frequency of one value per minute. The size of a chunk is determined by the Device Data Recorder.
  • In a query for continuously measured data that results in multiple chunks, each chunk’s Observation.effectivePeriod MUST be of the fixed chunk size. The last chunk MAY hold less data than fits the chunk. In this case, the Observation.status of the chunk MUST be set to preliminary (see Retrieving Data for details). A DiGA MAY use the logical id of the last chunk to request this chunk again later to obtain the missing data.
  • The only exceptions to the fixed chunk size are changes to the calibration.state or changes to the personal health device. In these cases, a chunk’s Observation.effectivePeriod can be shorter than the fixed chunk size (see Retrieving Data for details). The status of the chunk MUST be set to final in this case.

Details and examples on how to interpret and process chunks of continuously measured data are described in chapter Retrieving Data.

Examples

Example Search Queries

Most commonly, a GET or POST request will be performed to get multiple Observation resources. Additional search parameters MAY be specified to constrain which Observation resources SHOULD be returned. Common use cases include:

Request Explanation
GET /Observation Request all Observation resources available for the current patient and the authorized MIV (information about patient and MIV are associated with the Access Token).
GET /Observation?date=ge2025-07-23 Request all Observation resources since July 23rd for the current patient and the authorized MIV.
GET /Observation?code=http://loinc.org\|2339-0&date=ge2025-07-23 Request all Observation resources since July 23rd for the current patient and the authorized MIV that hold blood sugar values in mg/dl. If the LOINC code 2339-0 is not included in the MIV, the Device Data Recorder will return an empty searchset Bundle.
GET /Observation?_include=Observation:device The resulting searchset Bundle also includes the instances of the DeviceMetric resources referenced by the Observation resources.
GET /Observation?_include=Observation:device&_include:iterate=DeviceMetric:source The resulting Bundle includes all DeviceMetric and Device resources directly or indirectly linked with the contained Observation resources.
GET /Observation?_sort=-date&_count=1 Get the latest Observation (in the case of a continuous measurement, the current chunk) for the current patient and the authorized MIV.

Example: FHIR-READ

Request: GET /Observation/example-blood-glucose-measurement-1

Description: With FHIR-read interactions, a client can access a single resource instance by querying its internal ID. Restrictions by the OAuth scopes apply—if the client is not allowed to read this resource, a 404 error will be returned.

Response: Returned object is a single Observation resource.

  "resourceType" : "Observation",
  "id" : "example-blood-glucose-measurement-1",
  "meta" : {
    "profile" : [
      🔗 "https://gematik.de/fhir/hddt/StructureDefinition/hddt-blood-glucose-measurement"
    ]
  },
  "status" : "final",
  "code" : {
    "coding" : [
      {
        "system" : "http://loinc.org",

      }

  },

    "value" : 120,

    🔗 "reference" : "DeviceMetric/example-glucometer-metric"
  }
}
